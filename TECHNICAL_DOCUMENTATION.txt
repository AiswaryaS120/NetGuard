================================================================================
  1. PROJECT OVERVIEW & ARCHITECTURE
================================================================================

NetGuard is a real-time network intrusion detection system (IDS) that:
  - Captures live network packets from the machine's network interface
  - Extracts 14 statistical features from each packet's context using
    O(1) amortized sliding-window data structures
  - Runs a dual-model ML pipeline:
      Stage 1: Random Forest (known attack classification)
      Stage 2: Isolation Forest (zero-day anomaly detection)
  - Applies rule-based detection for SYN floods, DDoS, and port scans
  - Displays everything in a real-time dashboard with threat levels

ARCHITECTURE (Data Flow):

  ┌──────────────┐     ┌─────────────────────────────────────────────────┐
  │  Network     │     │              network_engine.py                  │
  │  Interface   │────>│                                                 │
  │  (Wi-Fi/LAN) │     │  PRODUCER (packet_callback)                     │
  └──────────────┘     │    │  Ultra-fast: only queues raw packets       │
                       │    v                                            │
                       │  _raw_queue (bounded, maxsize=10000)            │
                       │    │                                            │
                       │    v                                            │
                       │  CONSUMER (_consumer_worker) — separate thread  │
                       │    ├─ TrafficMonitor (14 features, O(1))        │
                       │    ├─ ml_engine (RF + Isolation Forest)         │
                       │    └─ rule_engine (SYN/DDoS/Scan rules)         │
                       │    │                                            │
                       └────┼────────────────────────────────────────────┘
                            v
                       ┌──────────────────┐     ┌────────────────────┐
                       │    data_queue     │────>│  gui_dashboard.py  │
                       │  (thread-safe    │     │  (reads queue,     │
                       │   Python Queue)  │     │   updates UI,      │
                       └──────────────────┘     │   shows alerts)    │
                                                └────────────────────┘

  main.py orchestrates everything:
    1. Creates a shared Queue (data_queue)
    2. Initializes ML engine (AnomalyDetector) and Rule engine (LogicEngine)
    3. Creates a SnifferManager that controls SnifferThread
    4. Creates the GUI (NetGuardDashboard) with start/stop callbacks
    5. The GUI's mainloop runs; user clicks "Start Monitoring"
    6. SnifferThread uses a producer-consumer pattern:
       - PRODUCER (Scapy sniff callback): captures packets and queues them
         instantly into _raw_queue — never blocks the capture loop
       - CONSUMER (background thread): pulls from _raw_queue, extracts
         14 features, runs ML (RF + IF) + rules, puts enriched data
         into data_queue
    7. Dashboard reads from data_queue every 500ms -> updates charts,
       alerts, threat level

TrafficMonitor CLASS (lines 9-113):
  Maintains O(1) amortized sliding-window statistics about network
  traffic to compute the 14 features. This is stateful — each packet
  updates the internal counters, and the features returned depend on
  recent history within a 2-second time window.

  __init__(self):
    self.lock = threading.Lock()
      A mutex to protect shared state from concurrent access.

    --- SOURCE IP TRACKING ---
    self.src_history = defaultdict(deque)
      Maps source IP -> deque of (timestamp, service) tuples.
      Used to compute count, srv_count, same_srv_rate, diff_srv_rate.

    self.src_service_counts = defaultdict(lambda: defaultdict(int))
      Maps source IP -> service -> count. O(1) lookup for service
      counts instead of iterating the entire history list.

    --- DESTINATION HOST TRACKING ---
    self.dst_history = defaultdict(deque)
      Maps destination IP -> deque of packet dicts containing
      {time, src, service, is_syn_err}. Used for all dst_host_*
      features.

    self.dst_host_count = defaultdict(int)
      Maps destination IP -> total packet count in the 2s window.

    self.dst_host_srv_count = defaultdict(lambda: defaultdict(int))
      Maps dst -> service -> count. For dst_host_srv_count feature.

    self.dst_host_src_count = defaultdict(lambda: defaultdict(int))
      Maps dst -> src -> count. For dst_host_same_src_port_rate.

    self.dst_syn_err = defaultdict(int)
      Maps dst -> SYN error count. For dst_host_serror_rate.

    self.dst_srv_syn_err = defaultdict(lambda: defaultdict(int))
      Maps dst -> service -> SYN error count. For
      dst_host_srv_serror_rate.

  update_and_get_features(self, packet_info) [lines 26-113]:
    This is the feature extraction engine. Called once per packet.
    Operates in three phases:

    PHASE 1: EVICTION (O(1) amortized) [lines 37-66]
      Removes entries older than 2 seconds from both source and
      destination deques. As each old entry is popped, its
      corresponding counters are decremented. Cleanup of empty
      counter keys prevents memory leaks.

    PHASE 2: ADDITION [lines 68-83]
      Appends the current packet to source and destination deques.
      Increments all relevant counters. Tracks SYN errors
      (SYN flag present, ACK flag absent) for serror_rate features.

    PHASE 3: O(1) FEATURE CALCULATION [lines 85-113]
      All 14 features are computed from pre-maintained counters,
      requiring no iteration over history. Division-by-zero is
      guarded throughout.

    THE 14 FEATURES:

    Feature  1: f_duration = 0
      Connection duration. Set to 0 because live packet capture sees
      individual packets, not complete TCP sessions.

    Feature  2: f_src_bytes = packet_info['length']
      Bytes sent from source. Uses the captured packet's length.

    Feature  3: f_dst_bytes = 0
      Bytes from destination. Set to 0 (only outbound visible).

    Feature  4: f_count = len(src_q)
      Number of connections from the same source IP in the last 2
      seconds. THE most important feature for detecting flood attacks.

    Feature  5: f_srv_count = src_srv_counts.get(service, 0)
      Number of connections to the SAME service (port) from this
      source in the last 2 seconds. O(1) dict lookup.

    Feature  6: f_same_srv_rate = srv_count / count
      Proportion of connections going to the same service.

    Feature  7: f_diff_srv_rate = (count - srv_count) / count
      Complement of same_srv_rate. High = scanning many services.

    Feature  8: f_dst_host_count
      Number of connections to the same DESTINATION host in the
      2-second time window. Maintained via counter dict.

    Feature  9: f_dst_host_srv_count
      Connections to same destination host AND same service.

    Feature 10: f_dst_host_same_srv_rate
      Ratio: dst_host_srv_count / dst_host_count

    Feature 11: f_dst_host_diff_srv_rate
      Ratio: (dst_host_count - dst_host_srv_count) / dst_host_count

    Feature 12: f_dst_host_same_src_port_rate
      Ratio of connections from the same source IP to this dst host.

    Feature 13: f_dst_host_serror_rate
      SYN error rate to this host. COMPUTED from tracked SYN errors
      (dst_syn_err[dst] / dst_host_count). Not hardcoded.

    Feature 14: f_dst_host_srv_serror_rate
      Service-level SYN error rate. COMPUTED per-service
      (dst_srv_syn_err[dst][service] / dst_host_srv_count).

    Lines 105-113: Returns the 14 features as a Python list, in the
    exact order expected by the trained model.

check_packet(self, packet_data) [lines 76-225]:
  The main entry point. Called once per packet.

  STEP 1: TIME WINDOW MANAGEMENT (lines 92-130)
    Every second (when elapsed >= 1.0), the engine:
    a) Evaluates the completed second for DoS patterns
    b) Commits current_syn and current_pkt to history deques
    c) Fills missed seconds with zeros (handles gaps)
    d) Resets current-second counters

    The dos_consecutive counter tracks how many consecutive seconds
    exceeded the DoS threshold. This prevents one-off spikes from
    triggering alerts.

  STEP 2: COUNTER UPDATES (lines 132-142)
    Increments current_pkt, adds flow to current_flows set.
    Detects pure SYN packets: 'S' in flag AND 'A' not in flag.
    This distinguishes SYN (connection initiation) from SYN-ACK
    (normal response).

  STEP 3: ANOMALY DETECTION (lines 144-225)
    Detection order: C (Port Scan) -> A (SYN Flood) -> B (DDoS)
    Port scans are evaluated FIRST because a port scan that uses
    SYN packets would also trigger SYN flood detection. If a port
    scan is detected, SYN flood checks are SUPPRESSED.

    C. PORT SCAN DETECTION (lines 150-178):
      Only triggers on SYN packets (is_syn check) to non-web ports.
      Web ports (80, 443, 8080, 8443) are excluded to prevent
      false positives from normal browser tab activity.
      Tracks unique destination ports per (src, dst) pair.
      If unique ports > SCAN_THRESHOLD (20), fires alert.
      Lines 174-178: Also suppresses SYN flood check if a scan
      alert was recently fired for this IP (within ALERT_COOLDOWN).

    A. SYN FLOOD DETECTION (lines 180-194):
      ONLY runs if is_port_scan is False (suppressed during scans).
      Computes dynamic threshold using _calculate_threshold().
      If current_syn exceeds threshold AND it's a SYN packet,
      fires a SYN flood alert (with cooldown check).

    B. DDOS / VOLUME DETECTION (lines 196-223):
      Computes dynamic threshold for total packet volume.
      For web ports, threshold floor is raised to 300 (line 207)
      to avoid false positives from normal HTTP/HTTPS traffic.
      Applies TWO additional filters:
        1. diversity_ratio < 0.2 — fewer than 20% unique flows
           means the traffic is concentrated (not normal browsing)
        2. dos_consecutive >= 2 — must sustain for 2+ seconds
      Also differentiates DoS vs DDoS by counting active IPs
      (>10 active IPs = DDoS, otherwise DoS).

_calculate_threshold(self, history, def_mean, def_std, k, min_floor):

    Threshold = Mean + K * StdDev

  If history has <2 samples, uses defaults. Always returns at least
  min_floor (safety floor). This is adaptive — the threshold adjusts
  to each IP's normal traffic pattern.

================================================================================
  4. METRICS — DEFINITIONS AND RESULTS
================================================================================

ACCURACY = (TP + TN) / (TP + TN + FP + FN)
  Overall correct predictions.

PRECISION = TP / (TP + FP)
  "Of predicted attacks, how many were real?"

RECALL = TP / (TP + FN)
  "Of actual attacks, how many were caught?"

F1 = 2 * (Precision * Recall) / (Precision + Recall)
  Harmonic mean — balanced metric.

TRAINING RESULTS (KDDTrain+.txt, 125,973 samples):
  5-Fold CV Accuracy:  99.76% (+/- 0.04%)
  Training Accuracy:   99.92%

TEST RESULTS (KDDTest+.txt, 22,544 samples):
  Accuracy:   77.02%
  Precision:  96.89%
  Recall:     61.61%
  F1 Score:   75.32%

  Confusion Matrix:
                    Predicted Normal   Predicted Attack
  Actual Normal        9,457              254
  Actual Attack        4,927             7,906

NOTE: The 77% test accuracy is a known property of NSL-KDD. The test
set contains 17 NOVEL attack types not present in training data. The
4,927 false negatives are entirely from these novel types. For known
attacks, accuracy is near 100%. Precision of 96.89% means almost no
false alarms in live monitoring.

================================================================================
  7. ML THEORY CONCEPTS
================================================================================

7.1 RANDOM FOREST
─────────────────
An ensemble of 300 decision trees, each trained on a random subset
of data (bootstrap sampling) and considering random feature subsets
at each split. Final prediction = majority vote.

Key properties:
  - Each tree sees ~63.2% of training samples (bootstrap)
  - At each split, considers sqrt(14) ≈ 4 random features
  - Split criterion: Gini impurity reduction
  - Gini(node) = 1 - SUM(p_i^2) for each class i
  - Pure node: Gini=0; 50/50 split: Gini=0.5

Why it works for IDS:
  - Non-parametric (no distribution assumptions)
  - Handles mixed scales without normalization
  - Built-in feature importance
  - Resistant to overfitting via ensemble averaging
  - Fast prediction (parallel tree traversal)


7.2 DECISION TREES
──────────────────
Each tree recursively partitions data with "If feature_j <= t" splits.
Optimal split maximizes:
  IG = Gini(parent) - [w_L * Gini(left) + w_R * Gini(right)]

Example splits in NetGuard:
  "src_bytes <= 105?"   — separates zero-payload SYN floods
  "count <= 45?"        — volume threshold for attack detection


7.3 STANDARDSCALER (Z-SCORE NORMALIZATION)
──────────────────────────────────────────
  z = (x - mean) / std
  Transforms each feature to zero mean, unit variance.
  Fitted on training data, applied identically at prediction time.
  Cost: 14 multiplications + 14 subtractions per packet.


7.4 CROSS-VALIDATION (5-FOLD)
─────────────────────────────
  Split data into 5 folds. For each fold:
    Train on 4/5, test on 1/5. Record accuracy.
  Report mean and std of 5 accuracies.
  Purpose: estimate generalization without using the test set.
  Result: 99.76% (+/- 0.04%) — very stable, no overfitting.


7.5 PRECISION-RECALL TRADE-OFF
──────────────────────────────
  Lower threshold -> more attacks caught (↑recall) but more
  false alarms (↓precision).
  Higher threshold -> fewer false alarms (↑precision) but some
  attacks missed (↓recall).
  NetGuard uses 0.45 (slightly aggressive) because:
    - 96.89% precision provides room
    - The noise filter is a second FP barrier
    - Missing attacks is worse than false alarms in security


7.6 BAGGING (BOOTSTRAP AGGREGATING)
───────────────────────────────────
  Train N models on random subsets, average predictions.
  Variance reduction: Var(ensemble) ≈ Var(single) / N
  300 trees reduces prediction variance by ~300x versus a single tree.


7.7 OVERFITTING CONTROL
───────────────────────
  max_depth=25: prevents trees from growing until each leaf is one sample
  min_samples_split=5: requires 5+ samples to create a new split
  min_samples_leaf=2: no single-sample leaves
  Ensemble: averaging 300 trees smooths individual tree errors


7.8 CLASS IMBALANCE
───────────────────
  class_weight='balanced' adjusts loss weights:
    weight_j = total_samples / (n_classes * class_j_count)
  For NetGuard: normal=0.935, attack=1.074
  Effect: model pays slightly more attention to attack samples.


7.9 NSL-KDD DATASET
────────────────────
  125,973 training records, 22,544 test records.
  41 features per record + 1 label.
  Feature groups: basic (9), content (13), time-based (9), host-based (10).
  17 novel attack types in test set not present in training.
  NetGuard uses 14 of the 41 features — those computable from live packets.


7.10 FEATURE IMPORTANCE (GINI IMPORTANCE)
─────────────────────────────────────────
  For each feature, sum the weighted Gini reduction across all nodes
  in all trees where that feature was used for splitting.
  Normalized so importances sum to 1.0.

  Top 5 in NetGuard:
    1. src_bytes               0.2833
    2. dst_bytes               0.1889
    3. same_srv_rate           0.0946
    4. dst_host_srv_count      0.0814
    5. dst_host_same_srv_rate  0.0563


7.11 PROBABILITY ESTIMATION IN RANDOM FOREST
─────────────────────────────────────────────
  predict_proba(X) = fraction of trees that voted each class.
  P(attack) = (# trees predicting "attack") / 300
  This gives a calibrated confidence score, not just a binary label.
  Higher P(attack) = more trees agreed it's malicious.

7.13 ISOLATION FOREST (UNSUPERVISED ANOMALY DETECTION)
──────────────────────────────────────────────────────
  An unsupervised ensemble method that detects anomalies by measuring
  how easily a data point can be isolated from the rest of the data.

  Algorithm:
    1. Build N isolation trees (iTrees), each on a random subsample.
    2. Each iTree recursively splits data by choosing a random feature
       and a random split value between min and max of that feature.
    3. Anomalies are isolated in fewer splits (shorter path length)
       because they are "few and different" from the majority.

  Key formula:
    Anomaly Score s(x, n) = 2^(-E(h(x)) / c(n))
    Where:
      h(x) = average path length for sample x across all trees
      c(n) = average path length for unsuccessful search in BST of n items
    Score close to 1.0 = anomaly, close to 0.5 = normal.

  In Scikit-learn:
    predict(X) returns 1 (inlier) or -1 (outlier)
    decision_function(X) returns anomaly score (more negative = more anomalous)

  Why it works for zero-day detection:
    - No labels needed — learns what "normal" looks like from training data
    - Novel attack types produce unusual feature combinations that are
      easy to isolate, even if the RF has never seen that attack pattern
    - Linear time complexity: O(n * t * log(ψ)) where t = trees, ψ = subsample
    - Low memory footprint compared to density-based methods

  NetGuard configuration:
    - Model file: iforest_model.pkl
    - Used as Stage 2 in ml_engine.py — only when RF says "normal"
    - Filters: count >= 50 AND anomaly_score < -0.05
    - Returns "Zero-Day Attack" when both filters pass
